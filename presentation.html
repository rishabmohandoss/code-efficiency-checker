Time Complexity

How cost explodes
as input grows

Each line represents how many operations your code performs as you give it more data. The steeper the curve, the worse the problem.

ops ↑ input size →
O(1) Best

Constant time. Doesn't matter how big the input is — same speed always. Think: HashMap lookup.

O(log n) Great

Logarithmic. Doubles input, adds one step. Binary search — cuts the problem in half each time.

O(n) Acceptable

Linear. Visit each item once. Usually fine — a simple loop reading through a list.

O(n²) Danger

Quadratic. A loop inside a loop. 1,000 items = 1 million operations. This is what we detect.

Unique Feature

Detecting AI Slop
in your codebase

LLMs generate code that looks correct but hides subtle anti-patterns — code that passes code review but silently breaks under real-world conditions.

{ }

Error Swallowing

Empty catch {} blocks that silently swallow failures. The program "works" — until it doesn't, with no trace why.

?.

Happy Path Bias

Missing null checks. AI writes for the case where everything works, not the edge cases that actually happen in production.

inf

Infinite Loop Traps

while(true) without a guaranteed exit. AI-generated retry logic is a common source of this.

#

Magic Numbers

Hardcoded 86400, 3.14159, 255 scattered everywhere. Unmaintainable and error-prone.

-- AI-Generated (flagged)

async function fetchUser(id) {
  try {
    const res = await api.get(id)
    return res.data.user.profile
  } catch (e) {
    pass  // ← swallows all errors
  }
  while(true) {   // ← no exit
    retry()
  }
}

++ What it should look like

async function fetchUser(id) {
  if (!id) throw Error("id required")

  const res = await api.get(id)
  if (!res?.data?.user) {
    throw Error("user not found")
  }
  return res.data.user.profile
}
