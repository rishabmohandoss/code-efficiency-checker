# Roadmap: Code Efficiency Checker

## Overview

This milestone takes an existing, shipped browser-based static analysis tool and systematically improves it: first cleaning up tech debt and fragile areas, then restructuring the codebase into maintainable modules with test coverage, then adding performance improvements, and finally layering in user-facing features (actionable fix suggestions, visualization, and learning tools) before shipping external integrations for GitHub Actions and VS Code.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 1: Codebase Health** - Remove dead code, fix silent errors, and patch fragile areas before new work begins
- [ ] **Phase 2: Structural Refactor + Tests** - Extract the analysis engine, services, and UI into dedicated modules and add test coverage for them
- [ ] **Phase 3: Performance** - Offload analysis to a Web Worker and lazy-load jsPDF so the UI never blocks
- [ ] **Phase 4: Fix Suggestions** - Show users corrected code snippets, side-by-side diffs, and real-world performance impact estimates alongside each flagged issue
- [ ] **Phase 5: Visualization** - Add a complexity growth chart and replace the plain textarea with a syntax-highlighted code editor
- [ ] **Phase 6: Learning Features** - Add an interactive fix challenge, jargon tooltips, and explanations for rules that passed
- [ ] **Phase 7: Integrations** - Ship a GitHub Actions workflow for PR analysis and a VS Code extension for in-editor analysis

## Phase Details

### Phase 1: Codebase Health
**Goal**: The codebase is clean and honest — no dead files, no silent failures, no known fragile behaviors that would corrupt future work
**Depends on**: Nothing (first phase)
**Requirements**: HLTH-01, HLTH-02, HLTH-03, HLTH-04, HLTH-05
**Success Criteria** (what must be TRUE):
  1. `src/App_old_flashy.jsx` does not exist in the repository
  2. When a rule test function throws, a console warning appears with the rule's ID — not silence
  3. Code containing strings like `"http://example.com"` analyzes without the URL being stripped as a comment
  4. Regex patterns in `performanceRules.js` and `aiSlopRules.js` are declared as module-level constants (not inline in test functions)
  5. `window.THREE` is only assigned when it is not already set — no unconditional global mutation
**Plans**: TBD

### Phase 2: Structural Refactor + Tests
**Goal**: The analysis engine, GitHub service, and PDF service each live in their own module, the App.jsx UI is split into sub-components, duplicated rule logic is consolidated, and the new modules are covered by unit and performance tests
**Depends on**: Phase 1
**Requirements**: REFR-01, REFR-02, REFR-03, REFR-04, REFR-05, TEST-01, TEST-02, TEST-03, TEST-04
**Success Criteria** (what must be TRUE):
  1. `src/engine/analyzer.js` exists and `src/App.jsx` calls it rather than containing `runAnalysis` inline
  2. `src/services/github.js` and `src/services/pdfExport.js` exist and `App.jsx` imports from them
  3. `InputPanel`, `ResultsPanel`, and `ExportButton` are distinct components that `App.jsx` composes
  4. The five similar "inside loop" rules are generated by a shared factory function rather than repeated definitions
  5. Unit tests pass for GitHub integration, `cleanLines` edge cases, and loop depth tracking; analysis completes under 100ms on 500-line files and under 500ms on 2000-line files
**Plans**: TBD
**UI hint**: yes

### Phase 3: Performance
**Goal**: The UI remains responsive during analysis of large files, and the initial page load does not pay the cost of loading jsPDF
**Depends on**: Phase 2
**Requirements**: PERF-01, PERF-02
**Success Criteria** (what must be TRUE):
  1. Pasting a 2000-line file and clicking Analyze does not freeze or block the browser UI thread
  2. The jsPDF library is not loaded on initial page load — it loads only when the user clicks "Export PDF"
**Plans**: TBD

### Phase 4: Fix Suggestions
**Goal**: Every flagged issue shows the user a corrected code snippet, a side-by-side diff of the problem vs the fix, and an estimate of the real-world performance cost
**Depends on**: Phase 2
**Requirements**: FIXS-01, FIXS-02, FIXS-03
**Success Criteria** (what must be TRUE):
  1. Each flagged issue displays a corrected code snippet next to the problematic code
  2. A side-by-side diff view shows original problematic code on the left and suggested fix on the right
  3. Each flagged issue includes a plain-language performance impact estimate (e.g., "This nested loop on 10k rows takes approximately 10 seconds")
**Plans**: TBD
**UI hint**: yes

### Phase 5: Visualization
**Goal**: Users can see how their detected complexity class scales at real input sizes, and they enter code into a syntax-highlighted editor rather than a plain textarea
**Depends on**: Phase 4
**Requirements**: VIZ-01, VIZ-02
**Success Criteria** (what must be TRUE):
  1. After analysis, a chart shows operation counts for the detected complexity class at N=100, N=1000, and N=10000
  2. The code input area is a syntax-highlighted editor (CodeMirror or Monaco Lite) — not a plain `<textarea>`
**Plans**: TBD
**UI hint**: yes

### Phase 6: Learning Features
**Goal**: Students can actively practice fixing code, quickly understand jargon without leaving the page, and learn why non-flagged rules passed
**Depends on**: Phase 4
**Requirements**: LRN-01, LRN-02, LRN-03
**Success Criteria** (what must be TRUE):
  1. The tool presents a broken code snippet as a challenge, accepts the user's attempted fix, and reports whether it passes all rules
  2. Hovering a jargon term like "O(n²)" or "linear scan" shows a plain-language popover explanation
  3. A "Why this passed" section is visible in results, with a brief explanation for each rule that did not trigger
**Plans**: TBD
**UI hint**: yes

### Phase 7: Integrations
**Goal**: Developers can analyze code without visiting the website — directly from GitHub PRs via GitHub Actions, or from their editor via a VS Code extension
**Depends on**: Phase 2
**Requirements**: INTG-01, INTG-02
**Success Criteria** (what must be TRUE):
  1. A GitHub Actions workflow file exists in the repo; when installed in a target project, it analyzes PRs and posts analysis results as a PR comment
  2. A VS Code extension command runs analysis on the currently open file and displays results in VS Code — without opening a browser
